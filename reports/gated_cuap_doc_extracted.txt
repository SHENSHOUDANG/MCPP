[Heading1] 1. 背景、失败诊断与本次目标
[Heading2] 1.1 固定 CUAP 的实验问题
现有固定 CUAP 采用固定 beta，将 Novelty、Frontier、Repeat 和 Conflict 形成的局部动作效用持续叠加到 actor logits。三模型消融结果表明，它在部分障碍压力场景中能够改善 Coverage-AUC 或重复访问，但整体收益不稳定，且部分场景的智能体间重叠和大地图多智能体迁移表现变差。
根本原因不是单一权重未调好，而是固定 CUAP 对所有状态、所有覆盖阶段和所有局部拓扑使用同一干预强度。局部启发式可能在开阔前期有效，却会在队友观测相似、残余补漏或必要跨越已覆盖区域时误导 actor。
[Heading2] 1.2 本次结构修正的唯一目的
[TABLE]
目标让策略自己学习“当前状态下是否应该相信 CUAP，以及相信多少”，而不是继续人工调多套 beta 或多套权重。
[/TABLE]
本次不是重新设计 CUAP，也不是增加新的任务分配模块。原始 CUAP 评分先保持不变，只将固定干预改为可学习标量门控，并取消危险的逐状态 z-score 放大。
[Heading2] 1.3 本次明确不做的改动
[ListBullet] 不引入区域分配、分层策略、异构图、反事实优势或新 critic。
[ListBullet] 不更改 depot 触发逻辑、返航奖励、任务终止条件。
[ListBullet] 不为不同地图、不同 agent 数量或不同阶段手工设置不同 CUAP 权重。
[ListBullet] 不删除固定 CUAP 表现不好的困难场景，也不在测试集上反复调参。
[Heading1] 2. 应用场景与 Dec-POMDP 边界
[Heading2] 2.1 具体应用场景
本项目对应“灾后开放区域快速搜索覆盖与返航”规划层任务。临时应急指挥车、移动基站或安全起降点作为 depot，多架同构无人机或移动机器人从 depot 同点/邻近位置出发。在没有完整先验地图的情况下，智能体依靠局部感知、显式记忆与有限范围通信搜索覆盖目标区域；覆盖达到阈值或收到基地阶段广播后进入返航阶段，并返回 depot。
该场景仅研究覆盖路径规划与协同决策，不宣称实现人员识别、目标检测、SLAM、连续动力学控制或完整灾后救援系统。栅格单元表示一个需要被传感器扫描/搜索的任务单元。

[Heading1] 3. 可学习门控 CUAP 的数学结构
[Heading2] 3.1 核心公式
gᵢ(t) = sigmoid(fψ(cᵢ(t))) ∈ [0,1]
所有 agent 共享同一个 fψ；cᵢ(t) 不包含 agent_id。
zᶠⁱⁿᵃˡᵢ(t) = zᵃᶜᵗᵒʳᵢ(t) + β · mᵖʰᵃˢᵉ(t) · gᵢ(t) · pᶜᵘᵃᵖᵢ(t)
mphase=1 表示覆盖阶段；返航阶段为 0。
门控为单个标量，形状为 [B,N,1]，仅决定“相信 CUAP 的程度”，不为四个动作分别输出 gate，避免门控网络变成第二个动作策略头。
[Heading2] 3.2 为什么必须是标量 gate
[TABLE]
设计 | 优点 | 风险/结论
标量 gate [B,N,1] | 只调节 CUAP 总强度；解释清晰；参数少；不改变 CUAP 动作排序 | 推荐 V1
动作级 gate [B,N,A] | 表达能力更强 | 等价于第二策略头，训练更不稳定，难判断创新来自 actor 还是 gate；禁止本轮使用
全局统一 gate | 实现简单 | 不能根据局部状态适配，退化成可学习 beta；不推荐
[/TABLE]
[Heading2] 3.3 参数初始化
门控网络最后一层权重初始化为 0，bias 初始化为 logit(0.1)≈-2.197，使所有样本初始 gate≈0.1。这样从 GAT-ON checkpoint 开始时，模型行为接近原基线，而不是立刻被 CUAP 接管。

[Heading1] 4. CUAP prior 的安全构造
[Heading2] 4.1 原始效用分数
Uᵢ(a)=wₙNᵢ(a)+w_fFᵢ(a)-w_rRᵢ(a)-w_cCᵢ(a)
第一轮门控实验保持当前 Novelty、Frontier、Repeat、Conflict 的定义与权重不变，避免同时改变评分和门控后无法归因。若现有固定 CUAP 已经实现上述四项，则直接复用原始分数计算函数。
[Heading2] 4.2 删除逐状态 z-score
[TABLE]
必须删除不得再使用 (scores-mean)/(std+eps)。当四个动作分数非常接近时，z-score 会把微小噪声放大为强动作偏好，这是固定 CUAP 可能恶化泛化的重要风险。
[/TABLE]
推荐只对合法动作求均值，然后使用有界的 center+tanh：
μᵥ = mean{Uᵢ(a) | a 为合法动作}
pᶜᵘᵃᵖᵢ(a)=tanh((Uᵢ(a)-μᵥ)/τ)
默认 τ=1.0。合法动作少于 2 个时，整个 cuap_prior 设为 0，applied_gate 也设为 0；非法动作的 prior 始终置 0，随后仍由 action mask 统一屏蔽。
[NoSpacing] def build_bounded_prior(raw_scores, action_mask, tau=1.0):
    valid = action_mask.bool()
    if valid.sum() <= 1:
        return torch.zeros_like(raw_scores)
    valid_mean = raw_scores[valid].mean()
    prior = torch.tanh((raw_scores - valid_mean) / tau)
    return torch.where(valid, prior, torch.zeros_like(prior))

tau | 1.0 | 控制 prior 饱和程度；本轮不做大范围搜索。

[Heading1] 5. 门控上下文特征设计
推荐 gate_context 使用 9 个局部、无身份、可解释标量。所有比例必须裁剪到 [0,1]；CUAP 分数差异类特征也应按固定常数缩放后裁剪，训练与评估使用完全相同的归一化。
[TABLE]
序号 | 特征名 | 计算 | 范围 | 用途
1 | cuap_margin | 合法动作最高分-次高分 | [0,1] | CUAP 对首选动作的置信度；不足2个合法动作时为0
2 | cuap_spread | 合法动作 max-min | [0,1] | 四动作区分度；使用固定 scale 裁剪，不用样本 std 归一化
3 | local_uncovered_ratio | 局部已知未覆盖/已知可通行 | [0,1] | 当前附近是否仍适合局部新覆盖启发
4 | local_frontier_ratio | 局部 frontier/已知可通行 | [0,1] | 未知边界密度；仅来自本体显式记忆
5 | local_covered_ratio | 局部已覆盖/已知可通行 | [0,1] | 局部是否进入高覆盖/转移状态
6 | recent_new_rate | 最近K步新增覆盖次数/K | [0,1] | 当前行为是否持续有效；K默认10
7 | stagnation_ratio | min(连续无新增步数/20,1) | [0,1] | 是否停滞；不人为规定 gate 增减方向
8 | neighbor_density | 通信邻居数/max(num_agents-1,1) | [0,1] | 邻居多时局部启发式可能导致共同吸引
9 | intent_conflict_mean | 四候选动作 conflict 均值或最大值 | [0,1] | 当前动作空间总体意图冲突程度
[/TABLE]

[ListBullet] agent_id、agent 顺序位置、固定角色标签。

[Heading2] 6.2 ActorCritic 内部新增共享 CUAPGate
门控网络必须注册为 ActorCritic 的子模块，确保进入 model.parameters()、state_dict 和 checkpoint。所有策略分布入口必须调用同一个 _policy_logits 方法，避免 act_batch 与 evaluate_actions 公式不一致。
[NoSpacing] class CUAPGate(nn.Module):
    def __init__(self, input_dim: int, hidden_dim: int = 32, init_prob: float = 0.1):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(input_dim, hidden_dim),
            nn.Tanh(),
            nn.Linear(hidden_dim, 1),
        )
        nn.init.zeros_(self.net[-1].weight)
        nn.init.constant_(self.net[-1].bias, math.log(init_prob / (1-init_prob)))

    def forward(self, context: torch.Tensor) -> torch.Tensor:
        return torch.sigmoid(self.net(context))
[NoSpacing] def _policy_logits(
    self,
    observations,
    neighbor_mask=None,
    edge_features=None,
    node_messages=None,
    cuap_prior=None,
    gate_context=None,
    phase_mask=None,
):
    features = self._actor_features(
        observations,
        neighbor_mask=neighbor_mask,
        edge_features=edge_features,
        node_messages=node_messages,
    )
    actor_logits = self.actor(features)

    if not self.use_gated_cuap:
        return actor_logits, None

    if cuap_prior is None or gate_context is None or phase_mask is None:
        raise ValueError('gated CUAP inputs are required')

    gate = self.cuap_gate(gate_context)              # [..., 1]
    applied_gate = gate * phase_mask                 # return phase -> 0
    final_logits = actor_logits + self.cuap_beta * applied_gate * cuap_prior
    return final_logits, applied_gate

[Heading2] 8.1 Rollout 的正确顺序
[ListNumber] 读取当前 observation、state、neighbor_mask、node_messages、action_mask。
[ListNumber] 在 env.step 之前调用 build_cuap_step_inputs，保证 prior/context 与当前 observation 同步。
[ListNumber] 将 prior、gate_context、phase_mask 转成 torch tensor，传入 model.act_batch。
[ListNumber] 使用 final_logits 对应分布采样动作并保存 old_log_prob。
[ListNumber] 把 observation、action、old_log_prob、cuap_prior、gate_context、phase_mask、action_mask 等同时存入 rollout buffer。
[ListNumber] 执行 env.step，更新下一时刻状态。

[Heading2] 8.2 PPO update 的正确顺序
[TABLE]
关键机制update 中必须把 batch_gate_context 送入当前门控网络，重新得到 new_gate。只有这样 PPO policy loss 的梯度才会更新 gate 参数。cuap_prior 是环境侧确定的非学习输入，可以从 rollout buffer 直接读取。
[/TABLE]

[Heading2] 8.3 为什么不能存 gate 输出
若 rollout 时把 gate 数值存入 buffer，update 时直接使用该数值，则 gate 与当前门控网络参数之间没有计算图，PPO loss 无法对 gate 参数反向传播。正确做法是存 gate_context，在 update 中重新前向计算 gate。
[Heading2] 8.4 为什么不能 update 时重新计算 CUAP 环境特征
PPO update 已经离开原始环境时刻，显式记忆和邻居 intent 可能已经变化。重新调用当前 env 会造成 prior 与 old_log_prob 对应的行为策略不一致，甚至引入未来信息。必须使用 rollout 当时保存的 prior 和 context。
[Heading2] 8.5 Action mask 顺序
actor logits → 加入 β·gate·prior → action mask → Categorical
禁止在 mask 后再加 CUAP；禁止给非法动作保留非零 prior。若 action mask 全 false，沿用项目原有安全回退逻辑，但 CUAP prior 和 gate 必须置 0，并记录异常。
[Heading1] 9. 从 GAT-ON checkpoint 初始化与优化器策略
[Heading2] 9.1 加载模型
使用训练最稳定的 GAT-ON checkpoint 初始化，而不是固定 CUAP checkpoint。固定 CUAP actor 可能已经学会抵消错误偏置，会污染门控实验。

[Heading2] 9.2 优化器
推荐新建优化器，不加载旧 optimizer state，因为参数集合已经增加门控网络。V1 可使用单优化器，学习率设为原微调学习率；若 GAT-ON 性能迅速崩塌，再使用参数组降低已有 actor/GAT 的学习率。
[NoSpacing] optimizer = torch.optim.Adam([
    {'params': actor_and_gat_params, 'lr': base_lr * 0.5},
    {'params': model.cuap_gate.parameters(), 'lr': base_lr},
    {'params': critic_params, 'lr': critic_lr},
])
[Heading2] 9.3 不建议冻结 actor
门控和 actor 最终共同决定策略，完全冻结 actor 可能限制适应。但若只做极短诊断，可先验证 gate 梯度，再进行联合微调。正式训练默认联合更新。
[Heading2] 9.4 gate 正则
第一版 gate_regularization=0，不增加人工监督或稀疏损失。只有 gate 快速饱和到 1 时，才考虑增加极小开启代价 λ·mean(gate)，建议 λ=1e-4，并作为额外消融而非默认。

[Heading2] 10.3 故障诊断表
[TABLE]
现象 | 优先检查 | 解释/处理
gate 始终≈0 | gate 梯度、update是否重算gate、初始化bias、优势是否正常 | 若实现无误且表现≈GAT-ON，说明 CUAP 无价值，停止投入
gate 很快≈1 | 是否仍有z-score、beta是否过大、gate LR、init_prob | 先确认实现；必要时降低beta或加极小开启代价

[ListNumber] Smoke test：小地图、短 rollout，所有单元测试通过，无 NaN。

[TABLE]

[Heading2] 11.5 明确停止条件
[ListBullet] gate 长期接近 0：策略主动拒绝 CUAP。结论是 CUAP 对现有 GAT-ON 没有额外价值，删除 CUAP。
[ListBullet] gate 长期接近 1，但覆盖/重叠仍不优于 GAT-ON：原始 CUAP 排序机制本身有问题，删除 CUAP。
[ListBullet] gate 随状态变化，但困难场景仍显著恶化：局部动作先验无法稳定转化为团队收益，删除 CUAP。
[ListBullet] 仅通过对不同测试场景设置不同 beta/权重才能变好：视为过拟合，不接受。
